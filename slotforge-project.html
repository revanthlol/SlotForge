Constraint-Driven Scheduling Engine

01 — Abstract

Project Abstract

Manual timetable construction in schools, colleges, and any organization that allocates people to time slots is a combinatorial problem routinely solved by hand through trial and error, despite being a textbook instance of Constraint Satisfaction (CSP) and Integer Linear Programming (ILP). As the number of agents, resources, and slots grows, the search space expands combinatorially, making manual scheduling slow, error-prone, and resistant to change whenever a single constraint shifts.

SlotForge, formulates resource-and-time scheduling as a constraint optimization problem and builds a production-grade multi-tenant platform around it. While the primary use case is academic timetabling — teachers, rooms, subjects, sections — the underlying model is domain-agnostic: any setup where finite resources must be assigned to agents across time slots without conflict (shift rosters, facility booking, exam scheduling) fits the same constraint structure, making the platform a general-purpose scheduler rather than a single-purpose timetable tool. Every entity is scoped under an organization_id from the outset, allowing multiple independent institutions or teams to run on the same system without data leakage between tenants.

Hard constraints — no agent or resource double-booked in the same slot, every section's weekly load satisfied, capacity and type compatibility respected — are encoded as boolean and linear constraints. Soft constraints — minimizing gaps, balancing daily load, respecting stated preferences — are stored as data rather than hardcoded, so the solver's behavior is configured by an admin through the interface, not by editing source code. Soft constraint outcomes are scored and surfaced per generation (preference satisfaction, utilization, gap minimization), giving every run a transparent quality signal rather than a black-box "done."

Because solver runs can take anywhere from a few seconds to over a minute on larger inputs, generation is handled as an asynchronous background job rather than a blocking request: the API enqueues a job onto a Redis queue, a dedicated solver worker process picks it up, and the result is written back to the database — letting a user leave and return, and letting multiple institutions generate timetables concurrently without one slow solve blocking another. The API and worker run as separate processes even when colocated on the same host, so a solver crash never takes the API down. Every generation is versioned and immutable once created; admins can compare versions, roll back, and explicitly promote a version from draft to published — end users only ever see the published schedule, never a half-reviewed draft. An audit log tracks every structural change (slot edits, room reassignments, staff deletions) for accountability in multi-admin settings.

The system separates concerns across three managed layers: Vercel hosts the React admin web app and the Flutter mobile client; an Oracle Cloud VPS runs the FastAPI service, the solver worker, Redis, and background task scheduling; and Supabase provides PostgreSQL, authentication, object storage (for exports, logos, and imports), and a path to realtime sync later. A premium tier layers an LLM-based review pass on top of the solved schedule and its constraint scores — translating the optimizer's trade-offs into plain-language explanations a non-technical admin can act on.

02 — Scope

Core Objectives

OBJ.01

Conflict-free generation

Zero double-bookings of agents, rooms, or resources across the generated timetable, verified, not assumed.

OBJ.02

Multi-tenant from day one

Every table scoped by organization_id — multiple institutions run on one system with zero data leakage.

OBJ.03

Async, non-blocking solve

Generation runs as a queued background job; users aren't stuck staring at a spinner for a 60-second solve.

OBJ.04

Versioned, draft → published

Every generation is immutable and comparable; end users only ever see a deliberately published version.

OBJ.05

Data-driven constraints

Constraints live in the database, not in code — admins configure solver behavior without a redeploy.

OBJ.06

AI-assisted review (premium)

An LLM pass over the solved schedule and its constraint scores, explained in plain language for non-technical admins.

04 — Implementation

Technology Stack

Layer Technology Rationale
Optimization Core
Solver PythonOR-Tools CP-SAT Industry-standard CSP/ILP solver; returns optimal solutions or proves infeasibility — essential for diagnosing unsolvable constraint sets.
Constraint storage DB-driven rules Constraints (max classes/day, unavailable slots, preferred rooms) live as rows in the database, not hardcoded constants — the solver builds itself from data.
Backend — API
API server FastAPI Async Python framework; enqueues solve jobs instead of blocking on them, returns immediately with a job id.
Job queue Redis Decouples request handling from solving. A 10–60s solve no longer ties up an API worker, and concurrent organizations can generate simultaneously.
Solver worker Python processRQ / Celery Runs as a separate systemd service from the API. A solver crash or long-running job never takes the API down.
Background scheduler APScheduler Nightly backups, cleanup jobs, and statistics generation — not urgent for MVP, slotted in early so the worker structure supports it later.
